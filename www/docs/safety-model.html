Safety Model

FlagLint applies only transformations it can prove are correct. This page explains how that proof works and what it means in practice.

AST-based analysis, not regular expressions

FlagLint parses your source files into an Abstract Syntax Tree (AST) using @typescript-eslint/typescript-estree before making any decision about flag usages.

This matters because regular expressions cannot reliably distinguish:

  • An aliased import (import ld from "launchdarkly-node-server-sdk") from a local variable named ld
  • A call on the LaunchDarkly client from a call on an object that happens to have a method named boolVariation
  • A dynamic key in a template literal from a static string constant
  • A require() call from a string that contains the word "require"

The AST-based scanner resolves the actual binding of each identifier before deciding whether it is a LaunchDarkly SDK call. This prevents false positives and false negatives that a text-search approach would produce.

The walker is iterative (not recursive), which means it handles arbitrarily deeply nested AST trees without stack overflow.

Clean git tree requirement for --apply

flaglint migrate --apply refuses to write to a dirty git working tree by default. This is not an arbitrary restriction: it ensures that every file modified by FlagLint appears clearly in your git diff and can be reviewed in isolation from unrelated changes.

If you need to override this (for example in a CI pipeline where changes are expected), pass --allow-dirty. When overriding, ensure the resulting diff is still reviewed before merge.

bash 
# Standard path — requires clean tree
flaglint migrate --apply

# Override — review the diff carefully before merging
flaglint migrate --apply --allow-dirty

No automatic rewrite of ambiguous patterns

FlagLint takes the position that an incorrect rewrite is worse than no rewrite. Anything that cannot be proven safe is reported for manual review, not guessed. The following patterns are never automatically rewritten:

  • Dynamic flag keys — If the flag key is a variable, template literal, or any expression that is not a static string literal, FlagLint cannot know the intended key and will not rewrite the call. It reports the call with the note "dynamic key — manual review required".
  • Detail method calls — boolVariationDetail, stringVariationDetail, numberVariationDetail, and jsonVariationDetail return a structured result object whose shape differs between the LaunchDarkly SDK and OpenFeature. These require manual review to update any code that consumes the detail result.
  • Bulk calls — allFlags() and allFlagsState() have no direct OpenFeature single-flag equivalent. They are inventoried but never auto-applied.
  • Unknown fallback type — If the type of the fallback value cannot be statically determined (e.g. a variable whose type is not known), FlagLint cannot select the correct OpenFeature typed getter and will not rewrite the call.
  • No proven client binding — --apply only rewrites files with a local OpenFeature.getClient() binding or a configured imported shared client allowlisted with openFeatureClientBindings.modulePatterns. If the binding is absent, unconfigured, or ambiguous, the file is skipped.

Required review and tests before merge

Even when FlagLint applies a transformation, it does not validate that the resulting code is behaviorally identical to the original. FlagLint is a static analysis tool — it does not run your application.

The recommended workflow after --apply:

  1. Run git diff and review every changed file
  2. Run your full test suite against the rewritten code
  3. Verify that the LaunchDarkly OpenFeature provider is initialized before any flag evaluation in your integration and end-to-end tests
  4. Open a pull request and require review before merging
⛔

Do not merge a --apply diff without running your test suite. FlagLint preserves call arguments structurally but cannot verify application-level behavior.

Idempotency

flaglint migrate --apply is idempotent. Running it twice on the same codebase — with the same analysis — produces no additional changes. Files that have already been migrated to openFeatureClient calls are recognized and skipped.

What FlagLint does not do

  • It does not evaluate feature flags at runtime
  • It does not connect to LaunchDarkly's API or streaming endpoints
  • It does not modify your LaunchDarkly dashboard or flag definitions
  • It does not remove LaunchDarkly packages from your package.json
  • It does not generate or modify test files
  • It does not make networking calls of any kind during a scan or apply
