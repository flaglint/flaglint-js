OpenFeature Provider Setup

A one-time manual bootstrap that wires the LaunchDarkly OpenFeature provider into your application. This must be complete before flaglint migrate --apply can rewrite any call sites.

ℹ️

FlagLint intentionally does not automate the provider bootstrap. It is application startup code that is specific to your deployment, initialization order, and context shape. flaglint migrate --dry-run includes this guidance inline as a reminder.

Step 1 — Install packages

Install the OpenFeature server SDK and the LaunchDarkly OpenFeature provider. Keep the LaunchDarkly Node.js server SDK package you already use — either @launchdarkly/node-server-sdk or legacy launchdarkly-node-server-sdk.

bash 
npm install @openfeature/server-sdk \
            @launchdarkly/node-server-sdk \
            @launchdarkly/openfeature-node-server
⚠️

Do not remove your LaunchDarkly Node.js SDK package. LaunchDarkly remains your feature flag provider. The OpenFeature provider package depends on the LaunchDarkly SDK at runtime.

Step 2 — Create a bootstrap file

Create a dedicated file (e.g. src/provider/setup.ts) that initializes the OpenFeature provider and exports the client. Keep all LaunchDarkly initialization in this one file — it will be the file you exclude from flaglint validate.

typescript 
import { LaunchDarklyProvider } from "@launchdarkly/openfeature-node-server";
import { OpenFeature } from "@openfeature/server-sdk";

const ldProvider = new LaunchDarklyProvider(process.env.LD_SDK_KEY!);
await OpenFeature.setProviderAndWait(ldProvider);

// Evaluation context needs a targeting key.
// Use OpenFeature `targetingKey` or keep an existing LaunchDarkly `key`:
// { targetingKey: user.id }  or  { key: user.id }

export const openFeatureClient = OpenFeature.getClient();

Step 3 — Import the client in service files

In each service file that will be migrated, import openFeatureClient from your bootstrap file. This is what flaglint migrate --apply checks for before rewriting any call in a file.

typescript 
import { openFeatureClient } from "../provider/setup";

// Now flaglint migrate --apply can rewrite ldClient calls in this file
const enabled = await openFeatureClient.getBooleanValue("checkout-v2", false, {
  targetingKey: user.id,
});

Evaluation context

OpenFeature uses targetingKey where LaunchDarkly uses key in context objects. The LaunchDarkly OpenFeature provider maps both, so existing { key: user.id } contexts will continue to work. For new code, prefer { targetingKey: user.id }.

LaunchDarkly context OpenFeature context
{ key: user.id } { targetingKey: user.id } (preferred) or { key: user.id }
{ kind: "user", key: user.id, name: user.name } { targetingKey: user.id, name: user.name }

Excluding the bootstrap file from validate

Because the bootstrap file itself contains provider initialization code, exclude it from the --no-direct-launchdarkly rule:

bash 
flaglint validate --no-direct-launchdarkly \
  --bootstrap-exclude src/provider/setup.ts
