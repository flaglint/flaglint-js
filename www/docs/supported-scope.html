Supported Scope

FlagLint's scope is intentionally narrow: accurate, safe automation on the call types where it can prove the transformation is correct. Everything outside that scope is inventoried and surfaced for manual review.

SDK scope

FlagLint currently supports the LaunchDarkly Node.js server-side SDK in both package forms: launchdarkly-node-server-sdk (legacy) and @launchdarkly/node-server-sdk (SDK v8+, scoped package). Both ESM import and CJS require() forms are recognized. Import aliases are resolved before matching.

Browser SDKs, React SDKs, Go, Java, Python, and other server SDKs are outside the current scope. Calls from those SDKs will not be detected or rewritten.

API coverage table

The table below shows every LaunchDarkly Node.js server SDK call pattern that FlagLint recognizes, and what it does with each.

LaunchDarkly call pattern Supported for scan Supported for apply Manual review Unsupported OpenFeature equivalent
ldClient.boolVariation(key, ctx, false) Yes Yes — — openFeatureClient.getBooleanValue(key, false, ctx)
ldClient.stringVariation(key, ctx, "") Yes Yes — — openFeatureClient.getStringValue(key, "", ctx)
ldClient.numberVariation(key, ctx, 0) Yes Yes — — openFeatureClient.getNumberValue(key, 0, ctx)
ldClient.jsonVariation(key, ctx, {}) Yes Yes — — openFeatureClient.getObjectValue(key, {}, ctx)
ldClient.boolVariationDetail(key, ctx, false) Yes No Yes — Detail result shape differs — manual review required
ldClient.stringVariationDetail(key, ctx, "") Yes No Yes — Detail result shape differs — manual review required
ldClient.numberVariationDetail(key, ctx, 0) Yes No Yes — Detail result shape differs — manual review required
ldClient.jsonVariationDetail(key, ctx, {}) Yes No Yes — Detail result shape differs — manual review required
Dynamic flag key: ldClient.boolVariation(flagKey, ctx, false) Yes No Yes — Key must be a static string literal for auto-apply
ldClient.allFlags(ctx) Yes No Yes — Bulk call — no single-flag codemod available
ldClient.allFlagsState(ctx) Yes No Yes — Bulk call — no single-flag codemod available
Unknown fallback type Yes No Yes — Fallback type must be statically determinable
LaunchDarkly browser SDK calls No No — Yes Out of scope — browser SDK not supported
LaunchDarkly React SDK calls No No — Yes Out of scope — React SDK not supported
Non-LaunchDarkly providers No No — Yes Out of scope — only LaunchDarkly Node.js server SDK

Column definitions

  • Supported for scan — Whether flaglint scan includes this call type in its report
  • Supported for apply — Whether flaglint migrate --apply can rewrite this call automatically after provider setup
  • Manual review — Patterns detected but not auto-applied are reported with file, line, and call type for manual resolution
  • Unsupported — Patterns outside the current scope are not detected and will not appear in reports

Custom wrapper functions

If your codebase wraps LaunchDarkly SDK calls in helper functions (e.g. flagPredicate(key, ctx)), you can register them in flaglint.config.json so the scanner detects them as flag usages:

json 
{
  "wrappers": ["flagPredicate", "useFlag"]
}

Wrapper functions are reported by flaglint scan but are always marked as manual review — --apply never rewrites them automatically.

Detection precision

FlagLint's scanner is validated against 120 deterministic benchmark cases within the supported LaunchDarkly Node.js server-side SDK scope. Detection is AST-based, not regex: client binding patterns, import aliases, and CJS require() forms are resolved before matching.

100% precision and recall are limited to those 120 tested cases and to the Node.js server-side SDK call patterns listed in the table above.
