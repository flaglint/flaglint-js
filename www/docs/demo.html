Enterprise Demo

A realistic checkout-service example for a platform team standardizing direct LaunchDarkly Node.js server SDK calls on OpenFeature while keeping LaunchDarkly as the provider.

Demo location

The committed demo lives in examples/enterprise-checkout-service. It includes migration-in-progress source files, a completed state, generated sample reports, and a GitHub Actions workflow example.

bash 
git clone https://github.com/flaglint/flaglint.git
cd flaglint
npm ci
npm run build
cd examples/enterprise-checkout-service

Walkthrough commands

bash 
node ../../dist/bin/flaglint.js scan ./src \
  --config ./.flaglintrc \
  --format markdown \
  --output generated-reports/inventory-report.md

node ../../dist/bin/flaglint.js migrate ./src \
  --config ./.flaglintrc \
  --dry-run

node ../../dist/bin/flaglint.js validate ./src \
  --no-direct-launchdarkly \
  --config ./.flaglintrc

node ../../dist/bin/flaglint.js validate ./after-complete \
  --no-direct-launchdarkly \
  --config ./.flaglintrc

Expected completed-state output

text 
✓ validate --no-direct-launchdarkly: no direct LaunchDarkly evaluation calls found.
  Scanned 5 file(s).

What the demo proves

  • Inventory: direct LaunchDarkly calls, dynamic keys, detail evaluations, and bulk calls are reported.
  • Migration planning: dry-run output separates automatable calls from manual-review cases.
  • Guarded apply: only files with a proven OpenFeature client binding can be rewritten.
  • Policy enforcement: validation fails while direct SDK calls remain and passes for the completed state.
  • Provider continuity: LaunchDarkly remains the provider through the OpenFeature provider bootstrap.
