Observability Guide

Observe feature-flag evaluations with OpenFeature and OpenTelemetry

Standardize evaluations once. Add consistent observability next. Once application code calls OpenFeature instead of a vendor SDK directly, evaluations performed through the configured OpenFeature client have a consistent hook lifecycle where application teams can attach telemetry.

Three distinct responsibilities

Before attaching telemetry to feature-flag evaluations, it helps to understand which tool is responsible for what. These three layers are independent and complementary.

🔍
FlagLint
Performs static source analysis — flag scans and migration plan generation. FlagLint identifies direct LaunchDarkly SDK call-sites and produces a reviewable plan for replacing them with OpenFeature calls. It does not run at evaluation time and does not emit telemetry signals.
🔌
OpenFeature
The CNCF-standard feature-flag evaluation API. OpenFeature defines a vendor-neutral client interface and a hook lifecycle that fires before and after each flag evaluation. LaunchDarkly remains the provider — OpenFeature is the evaluation boundary your application code calls.
📡
OpenTelemetry
The CNCF observability framework. OpenTelemetry can record feature-flag evaluation signals as span attributes or log records, giving platform teams a way to correlate flag decisions with runtime behavior in their existing observability pipeline.
What this guide is about. FlagLint helps you complete the source migration to OpenFeature. This page explains what becomes possible after that migration is done: a consistent place to register telemetry for evaluations performed through the configured OpenFeature client within an application boundary.

The vendor-neutral evaluation boundary

When application code calls the LaunchDarkly SDK directly, each call-site is coupled to the LaunchDarkly client object. Adding instrumentation means wrapping every call-site individually — or monkey-patching the SDK.

OpenFeature changes this. Once your application calls OpenFeature instead of the LaunchDarkly SDK, evaluations made through the configured OpenFeature client pass through the OpenFeature hook lifecycle. A platform team can register a hook at startup and observe those evaluations inside that application boundary.

OpenFeature creates a vendor-neutral evaluation boundary where platform teams can attach telemetry and governance. The OpenFeature hook registration point is separate from the underlying provider integration.

OpenTelemetry feature-flag semantic conventions

OpenTelemetry defines semantic conventions for feature-flag signals. These conventions describe how flag evaluation results should be recorded — as attributes on a feature_flag.evaluation span event, or as equivalent log record attributes — giving a common schema across teams and tools.

Development status. The OpenTelemetry feature-flag semantic conventions are currently marked Development status by the OpenTelemetry project. They are subject to change before stabilization. Check the official specification before adopting them in production instrumentation: opentelemetry.io/docs/specs/semconv/feature-flags/

The current Development-status specification defines a feature_flag.evaluation span event carrying these attributes:

  • feature_flag.key — the string key that identifies the flag
  • feature_flag.provider_name — the name of the provider that resolved the flag (e.g., "LaunchDarkly")
  • feature_flag.variant — a controlled variant identifier, where one is available and approved for export

These attribute names may change as the specification stabilizes. Review the OpenTelemetry changelog when upgrading instrumentation libraries. Do not use older deprecated names such as feature_flag.enabled or feature_flag.system — use the current names above.

Privacy: Avoid exporting user-identifying evaluation context, unrestricted flag result values, or sensitive targeting data. Prefer controlled variant information where appropriate, and follow the current official specification together with your internal data-handling policies.

Code example: hook boundary for telemetry

The following TypeScript example shows the bootstrap file pattern used by FlagLint's migration plan. It includes the correct location where a platform team would register an OpenTelemetry hook — before any evaluation calls are made. The telemetry hook itself is illustrative; the actual implementation depends on your OpenTelemetry SDK setup.

Optional post-migration integration — not generated by FlagLint. FlagLint does not install OpenTelemetry dependencies, register hooks, configure collectors, or emit runtime feature-flag telemetry.
platform/feature-flags.ts TypeScript 
/**
	 * platform/feature-flags.ts
	 *
	 * Bootstrap file: LaunchDarkly is registered as the OpenFeature provider.
	 * This module centralizes the LaunchDarkly OpenFeature provider integration.
	 * Application services evaluate flags through OpenFeature rather than direct
	 * LaunchDarkly Node.js server SDK evaluation calls.
 *
 * This file is excluded from `flaglint validate --no-direct-launchdarkly`
 * via the --bootstrap-exclude flag in flaglint.config.json.
 */

import { LaunchDarklyProvider } from "@launchdarkly/openfeature-node-server";
import { OpenFeature, type Hook, type HookContext, type EvaluationDetails } from "@openfeature/server-sdk";

// ── 1. Configure the LaunchDarkly provider ─────────────────────────
if (!process.env.LD_SDK_KEY) {
  throw new Error("LD_SDK_KEY environment variable is required");
}
const ldProvider = new LaunchDarklyProvider(process.env.LD_SDK_KEY);

// ── 2. Telemetry hook (illustrative) ───────────────────────────────
//
	// This is where a platform team would register OpenTelemetry
	// instrumentation. The `after` stage observes successful evaluations made
	// through this OpenFeature integration; `error` observes provider errors.
//
// NOTE: OpenTelemetry feature-flag semantic conventions are currently
// marked Development status. Attribute names may change before
// stabilization. See:
// https://opentelemetry.io/docs/specs/semconv/feature-flags/
//
// A real implementation would import a Tracer from the OpenTelemetry SDK
// and set span attributes rather than logging. The hook boundary itself
// does not change — only the instrumentation body.
const featureFlagTelemetryHook: Hook = {
  after<T>(hookCtx: HookContext<T>, details: EvaluationDetails<T>): void {
    // Register span attributes using the OpenTelemetry feature-flag
    // semantic conventions (Development status as of this writing):
    //   feature_flag.key           → hookCtx.flagKey
    //   feature_flag.provider_name → hookCtx.providerMetadata.name
	    //   feature_flag.variant       → approved controlled variant identifier
	    //
	    // Example (replace with your actual OpenTelemetry tracer setup):
	    // Emit only an approved, controlled variant identifier where available.
	    // Do not export unrestricted evaluation values or sensitive context.
	    //
	    //   const span = trace.getActiveSpan();
	    //   span?.setAttribute("feature_flag.key", hookCtx.flagKey);
	    //   span?.setAttribute("feature_flag.provider_name",
	    //                      hookCtx.providerMetadata.name);
	    //   span?.setAttribute("feature_flag.variant", approvedVariantId);
  },

  error<T>(hookCtx: HookContext<T>, err: Error): void {
    // Record the evaluation error in your observability pipeline.
    // Example:
    //   const span = trace.getActiveSpan();
    //   span?.recordException(err);
    //   span?.setAttribute("feature_flag.key", hookCtx.flagKey);
    },
};

// ── 3. Register provider and hooks, then export the client ─────────
await OpenFeature.setProviderAndWait(ldProvider);

// Hooks registered at the API level apply to all clients globally.
OpenFeature.addHooks(featureFlagTelemetryHook);

export const openFeatureClient = OpenFeature.getClient("checkout-platform");

export type EvaluationContext = {
  targetingKey: string;
  [key: string]: unknown;
};
src/checkout.ts (after migration) TypeScript 
/**
 * src/checkout.ts
 *
	 * After `flaglint migrate --apply`:
	 *   - No LaunchDarkly SDK imports remain in this file.
	 *   - All evaluation calls go through the OpenFeature client.
	 *   - The telemetry hook registered in platform/feature-flags.ts can
	 *     observe evaluations made through this imported OpenFeature client.
 */

import { openFeatureClient, type EvaluationContext } from "../platform/feature-flags.js";

interface User {
  id: string;
  email: string;
  plan: "free" | "pro" | "enterprise";
}

export async function isCheckoutV2Enabled(user: User): Promise<boolean> {
  const ctx: EvaluationContext = { targetingKey: user.id, email: user.email, plan: user.plan };
  return await openFeatureClient.getBooleanValue("checkout-v2", false, ctx);
  // ↑ The featureFlagTelemetryHook fires here automatically.
}

export async function getPaymentProvider(user: User): Promise<string> {
  const ctx: EvaluationContext = { targetingKey: user.id, plan: user.plan };
  return await openFeatureClient.getStringValue("payment-provider", "stripe", ctx);
}
Key point. No OpenTelemetry code is needed in checkout.ts. The hook registered in the bootstrap file can observe evaluations performed through the configured OpenFeature client within this application boundary. This is the architectural benefit of migrating to the OpenFeature evaluation boundary.

Where to register telemetry in the enterprise-checkout-service demo

The examples/enterprise-checkout-service demo bundled with FlagLint illustrates the before-and-after migration pattern. If you are extending that demo to explore OpenTelemetry integration, the correct insertion point is:

  • File: platform/feature-flags.ts — this is the bootstrap file where LaunchDarklyProvider is registered with OpenFeature.setProviderAndWait().
  • After the setProviderAndWait() call, before exporting openFeatureClient, call OpenFeature.addHooks(yourTelemetryHook).
  • No OpenTelemetry code is needed in src/checkout.ts or other service files that use the exported client. The hook can observe evaluations made through that client.

The demo does not implement OpenTelemetry instrumentation itself — doing so would require an OpenTelemetry SDK dependency and a running collector, which are out of scope for a static-analysis CLI example. The code example above and the hook boundary in platform/feature-flags.ts show exactly where that instrumentation belongs.
