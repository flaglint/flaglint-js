flaglint migrate

Analyze migration readiness, generate reviewable before/after diffs, and apply guarded transformations to replace direct LaunchDarkly Node.js SDK calls with OpenFeature.

Usage

bash 
flaglint migrate [dir] [options]

[dir] defaults to the current working directory when omitted.

Options

Option Default Description
--output <file> MIGRATION.md Write the migration plan to a file
--dry-run — Print reviewable before/after diffs to stdout; includes provider setup guidance when a diff needs it. No files are modified.
--apply — Apply automatable transformations in-place. Requires a clean git working tree.
--allow-dirty — Override the clean git tree requirement for --apply.
--config <path> auto-detect Path to a config file
--exclude-tests — Skip *.test.*, *.spec.*, __tests__/, tests/

Examples

Write migration plan to MIGRATION.md

bash 
flaglint migrate ./src

Preview diffs without modifying files

bash 
flaglint migrate --dry-run

Apply automatable transformations

bash 
flaglint migrate --apply

Apply on a dirty working tree (override guard)

bash 
flaglint migrate --apply --allow-dirty

Example transformation

--apply rewrites typed LaunchDarkly Node.js server SDK evaluation calls to OpenFeature equivalents. Flag key, fallback value, evaluation context, and await are preserved exactly.

Before — direct LaunchDarkly Node.js server SDK

typescript 
const enabled = await ldClient.boolVariation("checkout-v2", { key: user.id }, false);
const theme   = await ldClient.stringVariation("color-theme", { key: user.id }, "light");
const timeout = await ldClient.numberVariation("timeout-ms",  { key: user.id }, 5000);

After — OpenFeature via LaunchDarkly provider

typescript 
const enabled = await openFeatureClient.getBooleanValue("checkout-v2", false, { key: user.id });
	const theme   = await openFeatureClient.getStringValue("color-theme", "light", { key: user.id });
	const timeout = await openFeatureClient.getNumberValue("timeout-ms",  5000,    { key: user.id });
ℹ️

LaunchDarkly continues to serve the flags — only the call-site API changes. Your LD_SDK_KEY and all provider configuration remain untouched.

--apply safety contracts

FlagLint applies only transformations it can prove are safe. The following guarantees always hold when using --apply:

  • Clean tree required — Refuses to write on a dirty git working tree unless --allow-dirty is passed
  • Proven client binding — Only rewrites files that already contain a local OpenFeature.getClient() binding or a configured imported shared client allowlisted with openFeatureClientBindings.modulePatterns
  • No dynamic keys — Flag keys computed at runtime are never rewritten; they are reported for manual review
  • No detail methods — *VariationDetail calls are excluded; their result shapes differ and require manual review
  • No bulk calls — allFlags() and allFlagsState() are excluded
  • Argument preservation — Flag key, fallback value, evaluation context, and await are preserved exactly
  • Idempotent — Running --apply twice on the same codebase has no additional effect

See Safety Model for the full explanation of how FlagLint determines what is safe to apply automatically.

Provider setup prerequisite

Before running --apply, complete the one-time provider bootstrap. --dry-run includes inline guidance. See OpenFeature Provider Setup for details.
