The Problem

Migration is hard to do safely.

🔍
You don't know what you have

Direct ldClient.variation() calls are scattered across your codebase. Before migrating, you need an accurate inventory — which call types, which files, which are dynamic keys, which need manual review.

⚠️
Mechanical rewrites break things

The LaunchDarkly and OpenFeature APIs differ in argument order, context shape, and detail result format. A careless codemod loses fallback values, drops context, or silently breaks await. FlagLint only rewrites scoped, explicit call sites.

🚦
Migration has no finish line

Without enforcement, new direct LD calls creep back in after the migration. flaglint validate --no-direct-launchdarkly gives you a CI gate that exits 1 the moment a direct LD evaluation call appears outside your bootstrap file.

How It Works

Scan, migrate, enforce — in three steps

01
Inventory your codebase
$ flaglint scan ./src
JSON Markdown HTML SARIF

Generates an inventory report in JSON, Markdown, HTML, or SARIF. Know exactly what LaunchDarkly Node.js SDK calls exist, where they are, and which require manual review. Use validate --format sarif to emit policy-violation findings for GitHub Code Scanning.

02
Review and apply migration
$ flaglint migrate --dry-run

Generates reviewable before/after diffs and includes provider setup guidance when a diff needs it. When ready, --apply rewrites only automatable call-sites — preserving flag key, fallback, context, and await exactly. Detail methods, dynamic keys, and bulk calls require manual review.

03
Enforce in CI
$ flaglint validate \
  --no-direct-launchdarkly

A CI gate that exits 1 whenever a direct LaunchDarkly evaluation call appears outside your provider bootstrap file. Migration stays complete — no new LD calls can land unreviewed.

Built For

Built for platform teams standardizing feature-flag access

FlagLint helps teams using LaunchDarkly across Node.js services move application evaluation calls behind an OpenFeature boundary without changing providers on day one.

Inventory
Inventory direct SDK coupling.

Find direct LaunchDarkly Node.js server SDK evaluation calls by file, flag key, and call type before migration starts.

Review
Separate safe rewrites from manual review.

Typed static evaluations can be previewed for migration. Dynamic keys, detail evaluations, and bulk calls stay visible for human review.

Architecture
Support shared OpenFeature client architectures.

Configured shared client bindings let services import a platform-owned OpenFeature client instead of creating one in every file.

CI
Enforce the boundary in CI with SARIF annotations.

Use validate --format sarif to annotate direct SDK policy violations in pull requests.

Supported Scope

Focused scope for safe automation

✅
Automatically migrated

LaunchDarkly Node.js server-side evaluation calls in TypeScript and JavaScript where the flag key, value type, fallback, context, and OpenFeature client binding are explicit.

🔎
Reported for manual review

Dynamic keys, detail evaluations, bulk flag-state calls, browser SDKs, React usage, and ambiguous patterns are surfaced in reports instead of being rewritten automatically.

🔁
Provider stays the same

LaunchDarkly continues to serve flags. OpenFeature becomes the application-facing evaluation API used by your services and enforced in CI.

Features

Guarded migration to OpenFeature

🔍
AST-based scanning

Detects LaunchDarkly Node.js server SDK calls by AST, not grep. Resolves import aliases and CJS require forms.

📊
Four report formats

JSON for pipelines. Markdown for PRs. HTML for sharing. SARIF via validate --format sarif for GitHub Code Scanning policy enforcement.

🛡
Guarded apply

Rewrites only when an OpenFeature client binding is proven in the file. Never touches detail methods, dynamic keys, or bulk calls.

⚡
Manual-review flagging

Dynamic keys, detail methods, and unknown fallbacks are reported for manual review — never silently skipped or incorrectly rewritten.

🔄
Argument-preserving codemod

Flag key, fallback value, evaluation context, and await are preserved exactly. LaunchDarkly remains your provider.

🚦
CI enforcement

Exit 1 on any direct LD evaluation call outside the bootstrap file. Enforce migration completeness on every PR.

Install

Works with Node.js services using the LaunchDarkly server SDK

Install globally
$ npm install -g flaglint
Or use without installing
$ npx flaglint scan .

Supports Node.js 20+. Works with Express, NestJS, Fastify, or any Node.js TypeScript/JavaScript codebase using the LaunchDarkly Node.js server-side SDK (launchdarkly-node-server-sdk or @launchdarkly/node-server-sdk).
