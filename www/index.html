Problem

Direct SDK migrations fail when teams cannot see the real call sites.

Direct SDK coupling is hard to inventory

LaunchDarkly evaluation calls hide across services, wrappers, and old modules. You need file, line, flag key, method, and manual-review context before changing code.

Unsafe codemods risk behavior changes

OpenFeature and LaunchDarkly use different method names and argument order. A rewrite must preserve flag key, fallback, context, value type, and existing await behavior.

Migrations regress without CI policy

After the first migration, new direct SDK calls can return unless CI blocks them and annotates the exact source line.

Scan, migrate, enforce.

Supported scope

Focused automation, explicit manual review.

Auto-migrated
  • Static `boolVariation`, `stringVariation`, `numberVariation`, and `jsonVariation` calls.
  • Generic `variation` only when fallback type is proven.
  • Call sites with proven OpenFeature client bindings.
Detected / manual review
  • Dynamic keys.
  • Detail evaluations.
  • Bulk calls like `allFlagsState`.
  • Configured wrappers and ambiguous bindings.
Outside coverage
  • Browser SDKs and React SDKs.
  • Non-Node SDKs.
  • Non-LaunchDarkly providers.
  • Production stale-flag detection.

Safety

Local source analysis with guarded rewrites.

Local AST analysis

FlagLint parses local JavaScript and TypeScript source files. It does not execute your application.

No LaunchDarkly SDK key needed

Scan, migrate, and validate operate on source code. They do not need a LaunchDarkly API key or SDK key.

No source upload

The CLI runs locally in your checkout and does not upload application source to a FlagLint service.

CI enforcement

Block direct SDK calls and upload SARIF annotations.

Blocking workflow

name: FlagLint Policy
on: [pull_request]

jobs:
  validate:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      security-events: write
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: |
          npx flaglint validate ./src \
            --no-direct-launchdarkly \
            --format sarif \
            --output flaglint-validation.sarif
      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: flaglint-validation.sarif

SARIF annotations

validate --format sarif emits direct-SDK policy findings using rule id flaglint.direct-launchdarkly. Upload the SARIF file with GitHub Code Scanning so reviewers see exact file and line annotations.

Do not set continue-on-error: true on the blocking validation step. Use if: always() only on a later SARIF upload step.
